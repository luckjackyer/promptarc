# Product

## Register

product

## Users

PromptArc serves creators, marketers, indie builders, designers, and ordinary users who want to turn prompt ideas into useful AI-generated images. They may arrive from the gallery, a prompt detail page, or the generator itself. In the generator flow, their main context is task-focused creation: write or reuse a prompt, choose generation parameters, wait for output, inspect candidates, download or continue editing.

## Product Purpose

PromptArc helps users discover, refine, generate, and reuse image prompts. The generator should become the primary product workflow: users should understand what they can create, generate with confidence, compare results, inspect details, and keep moving without guessing whether the system is working.

Success looks like:

- A user can generate one or more images without confusion.
- Result quantity, queue status, errors, and completion are explicit.
- Generated images are visually central, not buried under controls.
- The detail view feels like an image creation workspace, not a generic modal.
- Gallery, generator, and future history/account features use one consistent product vocabulary.

## Brand Personality

Professional, focused, creative.

The interface should feel calm and capable rather than decorative. It can have a premium dark workspace mood, but only when that supports image creation. It should feel closer to Linear, Raycast, Figma, Apple product inspection, and Replicate-style AI creation than to a marketing landing page.

## Anti-references

PromptArc should not look like:

- A generic SaaS landing page with oversized hero copy.
- An admin dashboard with dense controls but weak image priority.
- A generic lightbox with buttons attached.
- A mixed visual system combining glass, neon, random gradients, heavy cards, and unrelated accents.
- A desktop layout merely scaled down for mobile.
- A UI with temporary-looking glyphs, mismatched button styles, or unclear states.

## Design Principles

1. Image first: once generation starts, the created image and its candidates are the primary object.
2. Tools recede: controls should feel precise, available, and quiet, not louder than the artwork.
3. State is visible: generating, queued, completed, failed, selected, disabled, and loading states must be clear.
4. Familiarity earns trust: use standard product UI affordances unless a different pattern clearly improves the task.
5. One system: navigation, composer, results, detail workspace, and future history/account views should share the same component vocabulary.

## Accessibility & Inclusion

Target WCAG AA for contrast, focus visibility, labels, and keyboard access. Users should be able to navigate composer controls, parameter panels, image candidates, and detail views with keyboard support. Motion should be short, purposeful, and compatible with reduced-motion preferences. Touch targets on mobile should be at least 40px, with 44px preferred for primary controls.
